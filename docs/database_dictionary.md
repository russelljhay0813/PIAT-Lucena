## Database Overview

The PIAT School Management System uses a SQLite database with **18 tables** that support the complete academic operations lifecycle: enrollment, curriculum management, grade tracking, attendance monitoring, academic record generation, and system administration. The database is initialized on server startup in `backend/db.js` using `CREATE TABLE IF NOT EXISTS` with subsequent `ALTER TABLE ADD COLUMN IF MISSING` migrations for dynamic schema extensions.

**Database Technology:** SQLite 5  
**Driver:** `sqlite3` (Node.js)  
**Database File:** `backend/bwest.db`  
**Schema Management:** Code-defined in `backend/db.js`

## Table: `programs`

**Description:** Stores academic program definitions (e.g., Diploma in Hospitality Services and Technology). Programs are the top-level academic offerings of PIAT.

| 1 | `id` | TEXT | NO | — | Primary key (UUID) |
| 2 | `studentId` | TEXT | NO | — | FK to `students(studentId)` |
| 3 | `subjectId` | TEXT | NO | — | FK to `subjects(id)` |

| 7 | `type` | TEXT | YES | NULL | Grade type: 'activity', 'quiz', 'exam', 'overall' |
| 8 | `component` | TEXT | YES | NULL | Grade component name (e.g., "Quiz 1") |
| 9 | `status` | TEXT | YES | 'draft' | Grade workflow status: 'draft', 'submitted', 'finalized' |
| 10 | `submittedAt` | INTEGER | NO | — | Unix epoch ms timestamp of submission |

**Notes:**
- Grade status transitions: `draft → submitted → finalized`. Faculty initially save as `draft`, then submit for locking.
- `period` and `type` are nullable; when both are NULL, the grade represents an overall grade.
- The grade finalization utility (`grade-finalization-utils.js`) computes final grades from component grades.

## Schema Migration Notes

The PIAT SMS uses a progressive migration strategy in `backend/db.js`. After the initial `CREATE TABLE IF NOT EXISTS` blocks, the following columns are added dynamically if missing:

| Table | Columns Added via Migration | Purpose |
|-------|-----------------------------|---------|
| `users` | `temporaryPassword`, `semester`, `academicYear`, `firstLoginAt`, `lastLoginAt`, `studentId` | Expands user schema for student role support and login tracking |
| `attendance` | `studentName`, `subjectCode`, `subjectTitle`, `facultyId`, `time`, `academicYear`, `semester`, `program`, `yearLevel`, `section` | Denormalizes attendance display data for performance |
| `students` | `firstLoginAt`, `lastLoginAt`, `placeOfBirth`, `barangay`, `parentRelationship` | Adds geographic and login tracking fields |
